JWT (JSON Web Token): o que é, como funciona e como usar em sistemas

O JWT, ou JSON Web Token, é um padrão aberto definido pela RFC 7519 amplamente utilizado para autenticação e troca de informações entre sistemas de forma segura e compacta. Ele permite que um servidor emita um token assinado digitalmente que pode ser verificado por qualquer parte que possua a chave pública, sem a necessidade de consultar um banco de dados a cada requisição.

Estrutura de um token JWT

Um token JWT é composto por três partes separadas por pontos (.), cada uma codificada em Base64URL:

  • Header: contém o tipo do token (JWT) e o algoritmo de assinatura utilizado, como HS256 (HMAC-SHA256) ou RS256 (RSA-SHA256).
  • Payload: contém os claims — informações sobre o usuário ou contexto, como sub (subject/ID do usuário), iat (issued at/data de emissão) e exp (expiration/data de expiração).
  • Signature: é a assinatura digital gerada combinando o header, o payload e uma chave secreta (ou par de chaves no caso de RSA). É o que garante que o token não foi adulterado.

Como o JWT é usado na autenticação

O fluxo típico de autenticação com JWT funciona assim: o usuário faz login com suas credenciais; o servidor valida e emite um token JWT assinado; o cliente armazena o token (geralmente no localStorage ou em um cookie HttpOnly) e o envia no header Authorization: Bearer <token> em cada requisição subsequente. O servidor então valida a assinatura do token sem precisar consultar o banco de dados, tornando o processo stateless e escalável.

Claims padrão do JWT

A especificação RFC 7519 define os seguintes claims registrados mais utilizados:

  • iss (Issuer): identifica quem emitiu o token — geralmente o domínio da API.
  • sub (Subject): identifica o principal sujeito do token, normalmente o ID do usuário.
  • aud (Audience): identifica os destinatários para os quais o JWT é destinado.
  • exp (Expiration Time): timestamp Unix indicando quando o token expira.
  • iat (Issued At): timestamp Unix de quando o token foi emitido.
  • nbf (Not Before): timestamp indicando antes de qual momento o token não é válido.
  • jti (JWT ID): identificador único do token, útil para evitar replay attacks.

Algoritmos de assinatura: HS256 vs RS256

O HS256 usa HMAC com SHA-256 e uma chave secreta simétrica — a mesma chave é usada para assinar e verificar. É simples, mas exige que tanto o emissor quanto o verificador compartilhem a chave secreta. Já o RS256 usa RSA com SHA-256 e um par de chaves assimétricas: a chave privada assina o token e a chave pública verifica. É o padrão recomendado para sistemas distribuídos, pois a chave pública pode ser compartilhada livremente sem comprometer a segurança.

Boas práticas de segurança com JWT

Ao implementar JWT em seus sistemas, siga estas práticas:

  • Defina sempre um tempo de expiração curto (exp) — entre 15 minutos e 1 hora para tokens de acesso.
  • Use refresh tokens de longa duração armazenados em cookies HttpOnly para renovar os access tokens sem exigir novo login.
  • Nunca armazene dados sensíveis no payload — ele é apenas codificado em Base64, não criptografado.
  • Prefira RS256 em arquiteturas de microsserviços para evitar compartilhar chaves secretas entre serviços.
  • Implemente uma blocklist de tokens revogados para lidar com logout forçado e tokens comprometidos.

O JWT Decoder do 4DevTools permite decodificar qualquer token JWT diretamente no navegador, sem enviar dados para servidores externos.
